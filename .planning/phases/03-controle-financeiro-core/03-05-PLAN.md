<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User form input → transacaoService | All 5+1 fields traverse this boundary; service validates before Supabase insert |
| Supabase RLS → TransactionSheet read | transaction prop comes from server data already filtered by RLS |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02 | Tampering | TransactionSheet handleSave (valor) | high | mitigate | UI-layer validation: cents > 0 check before calling service. Service also validates (defense in depth per T-03-02 in Plan 02). |
| T-03-03 | Tampering | TransactionSheet (tipo, tipo_pessoa) | medium | mitigate | Toggle UI enforces valid values: tipo can only be 'entrada' or 'saida' (two buttons), tipo_pessoa can only be 'PF', 'PJ', or null. Service-layer validation also fires. |
| T-03-04 | Tampering / XSS | TransactionSheet (descricao render) | medium | mitigate | descricao rendered as JSX text node ({error} and {descricao} in value prop) — never dangerouslySetInnerHTML. Enforced in acceptance criteria. |
| T-03-05 | Tampering | TransactionSheet (data field) | medium | mitigate | data input type="date" produces YYYY-MM-DD format natively. Service validates format before insert. |
| T-03-01 | Elevation of Privilege | TransactionSheet delete | high | mitigate | transacaoService.delete uses .eq('id', id) with RLS — cannot delete another user's transaction even with crafted id. RLS WHERE clause applies before DELETE executes. |
</threat_model>
