<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase PostgREST | All data requests use anon JWT; RLS enforces user isolation at DB level |
| auth.uid() in RLS policies | Trust anchor: policies grant access only to the row owner |
| SECURITY DEFINER function | delete_user() runs with elevated privileges — must validate auth.uid() is not null |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-1-01 | Information Disclosure | All 5 Supabase tables (RLS policies) | high | mitigate | Each table has ENABLE ROW LEVEL SECURITY + policy using (select auth.uid()) = user_id. Verified in Task 2 human checkpoint: RLS badge active + anonymous SELECT returns 0 rows |
| T-1-02 | Elevation of Privilege | Supabase anon key vs service_role key in browser | high | mitigate | Only VITE_SUPABASE_ANON_KEY is exposed in browser; service_role key is NEVER used in Phase 1 client code. delete_user() uses SECURITY DEFINER instead of service_role |
| T-1-04 | Repudiation | delete_user() SECURITY DEFINER function | high | mitigate | SECURITY DEFINER function deletes WHERE id = auth.uid() — only the authenticated user's own row; no admin parameter accepted. AlertDialog confirmation in Phase 1 Plan 05 adds UI gate |
| T-1-SC | Tampering | npm/pip/cargo installs | high | mitigate | Package Legitimacy Audit completed in RESEARCH.md — all packages verified with weekly downloads and source repos; no SLOP packages; no genuine SUS flags |
</threat_model>
